# Base Routes

`@rapidrest/service-core` ships six ready-made route classes for common operational concerns — admin actions, Prometheus metrics, OpenAPI docs,
service status, WebSocket push delivery, and ACL management. Each one is a plain class prefixed `Base` (`BaseAdminRoute`, `BaseMetricsRoute`,
`BaseOpenAPIRoute`, `BaseStatusRoute`, `BasePushRoute`, `BaseACLRoute`) that implements all of the behavior for its endpoints, but carries no
`@Route` decorator of its own.

That last part matters: `Server.start()` only wires up classes it discovers via `ClassLoader` that are decorated with `@Route` (see
[Auto-Discovery](/docs/core-concepts/auto-discovery)). None of the `Base*Route` classes qualify on their own, so none of them are exposed by simply
installing `service-core` — you (or your scaffold) must extend one, decorate the subclass with `@Route`, and export it from somewhere under `src/`.

| Base class | Endpoints it implements | Docs |
|---|---|---|
| `BaseAdminRoute` | Cache clearing, live log tail, release notes, restart | [Admin](/docs/base-routes/admin) |
| `BaseMetricsRoute` | Prometheus metrics | [Metrics](/docs/base-routes/metrics) |
| `BaseOpenAPIRoute` | Swagger UI, raw OpenAPI JSON/YAML | [OpenAPI](/docs/base-routes/openapi) |
| `BaseStatusRoute` | Service status metadata | [Status](/docs/base-routes/status) |
| `BasePushRoute` | WebSocket push notification delivery | [Push](/docs/base-routes/push) |
| `BaseACLRoute<T>` | CRUD management of Access Control List records | [ACL](/docs/base-routes/acl) |

## The CLI scaffold wires most of these up for you

Every project generated by `rapidrest generate server` already includes a thin subclass for `BaseAdminRoute`, `BaseMetricsRoute`,
`BaseOpenAPIRoute`, `BaseStatusRoute`, and `BasePushRoute` under `src/routes/`, e.g.:

```ts title="src/routes/StatusRoute.ts"
import { BaseStatusRoute, RouteDecorators } from "@rapidrest/service-core";
const { Route } = RouteDecorators;

@Route("/status")
export class StatusRoute extends BaseStatusRoute {}
```

Because these files are ordinary auto-discovered route classes, they behave exactly like any other route in your project — nothing distinguishes
them at runtime as "built-in." You can delete the file to remove the endpoint entirely, change the `@Route` base path, or add further decorators
(`@Auth`, `@Protect`, additional `trusted_roles`) directly on the subclass to lock it down further.

`BaseACLRoute` is the exception — it isn't generated by the scaffold, since it requires picking a model (`AccessControlListMongo` or
`AccessControlListSQL`) that depends on which database your project uses. See [ACL](/docs/base-routes/acl) to add it yourself.

## Using them without the CLI scaffold

If you're wiring `service-core` into a project by hand — or you deleted one of the generated files and want it back — the pattern is the same for
every base route: extend it, apply `@Route()` with whatever base path you want, and export the class from anywhere under your source tree.

```ts
import { BaseMetricsRoute, RouteDecorators } from "@rapidrest/service-core";
const { Route } = RouteDecorators;

@Route("/metrics")
export class MetricsRoute extends BaseMetricsRoute {}
```

No further registration is required. `Server.start()` picks it up the same way it picks up any other `@Route`-decorated class.

## Why opt-in?

Earlier versions of the framework registered these routes automatically the moment `service-core` was loaded. In practice that meant every
deployment shipped with `/admin`, `/metrics`, `/openapi` and `/status` exposed at fixed paths whether or not that was desired — surprising for
services that wanted `/metrics` reachable only from an internal network, wanted `/admin` gated behind a stricter role than `trusted_roles`, or didn't
want an admin surface at all. Making registration explicit means the set of exposed endpoints for a given service is always visible by reading its
`src/routes/` directory, not by knowing which package versions happen to be installed.
